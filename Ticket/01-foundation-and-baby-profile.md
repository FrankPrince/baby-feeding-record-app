# Ticket 01 - 小程序基础工程、本地数据底座与宝宝档案

## 目标

搭建微信原生小程序第一版的基础工程结构，完成本地存储抽象、通用数据模型和单宝宝档案能力，为后续喂养、大小便、黄疸、首页和趋势模块提供稳定的数据底座。

## 开发范围

- 创建 `miniprogram/` 基础目录和底部 Tab 页面占位：
  - 今日
  - 记录
  - 趋势
  - 宝宝
- 建立分层目录：
  - `models/`
  - `services/`
  - `repositories/`
  - `rules/`
  - `utils/`
  - `components/`
- 实现本地存储基础设施：
  - `storageKeys`
  - `localRepository`
  - 单对象读写
  - 数组记录读写
  - 软删除字段预留
  - schemaVersion / syncStatus 等未来同步字段预留
- 定义核心模型默认结构：
  - `BabyProfile`
  - `BaseRecord`
  - `FeedingRecord`
  - `DiaperRecord`
  - `JaundiceRecord`
- 实现宝宝档案能力：
  - 创建宝宝档案
  - 编辑宝宝档案
  - 选择宝宝头像并持久化到小程序本地文件
  - 读取宝宝档案
  - 计算日龄和出生后小时数所需的基础工具
- 实现宝宝页 UI：
  - 头像
  - 昵称
  - 出生日期
  - 出生时间
  - 当前体重 kg
  - 出生孕周
  - 性别
  - 出生体重 kg
  - 是否早产
  - 是否存在黄疸高风险因素
  - 备注

## 不包含

- 喂养、大小便、黄疸记录的完整业务表单。
- 首页真实统计聚合。
- 趋势图绘制。
- 登录、云同步、多宝宝管理。
- 宝宝头像上传到云端或服务端。

## 技术要求

- 页面层只调用 service，不直接操作 `wx.setStorage` / `wx.getStorage`。
- repository 层统一封装微信本地存储，方便未来替换为云端 repository。
- 所有记录模型必须保留：
  - `id`
  - `localId`
  - `babyId`
  - `createdAt`
  - `updatedAt`
  - `deletedAt`
  - `syncStatus`
  - `createdBy`
  - `schemaVersion`
- 第一版只允许存在一个宝宝档案，但字段中保留 `babyId`。

## 验收标准

- 新安装打开小程序时，底部 Tab 显示今日、记录、趋势、宝宝 4 个入口。
- 首次进入宝宝页时，可以创建一个宝宝档案。
- 宝宝档案必填字段包含昵称、出生日期、出生时间、当前体重、出生孕周。
- 宝宝档案可选字段包含头像、性别、出生体重、是否早产、是否存在黄疸高风险因素、备注。
- 宝宝页顶部头像按钮可唤起微信头像选择能力。
- 保存宝宝档案后，关闭并重新打开小程序，已保存数据仍然存在。
- 再次进入宝宝页时，可以编辑并保存已有宝宝档案。
- 缺少出生日期或出生时间时，service 能返回“无法计算日龄 / 出生后小时数”的状态。
- 缺少当前体重时，service 能返回“无法估算奶量状态”的基础状态。
- 缺少出生孕周时，service 能返回“无法判断黄疸风险”的基础状态。
- 本地存储 key 至少包含：
  - `baby_profile`
  - `feeding_records`
  - `diaper_records`
  - `jaundice_records`
  - `app_meta`
- 代码中不包含登录、云同步、多成员共享或提醒功能的真实实现。

## 独立测试方式

- 使用微信开发者工具启动小程序，只测试宝宝页和底部 Tab 是否可用。
- 使用 mock `wx` storage 单独测试 repository：
  - 写入宝宝档案
  - 读取宝宝档案
  - 更新宝宝档案
  - 清空后返回空状态
- 使用固定出生日期、出生时间测试日龄和出生后小时数工具函数。
- 该 Ticket 不依赖喂养、大小便、黄疸模块完成；后续模块可通过 mock baby profile 独立开发。
